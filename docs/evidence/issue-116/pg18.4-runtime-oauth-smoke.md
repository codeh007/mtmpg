# Issue #116：PG18.4 runtime与OAuth smoke证据

本证据于2026-07-17T09:33:54Z（UTC）采集。真实OAuth gate的tracked实现提交为
`696a1cb9dc3e2416726c7cd88fbdaa68e6293c76`；采集后工作树为空。验证目标是精确
`postgres:18.4-bookworm@sha256:1961f96e6029a02c3812d7cb329a3b03a3ac2bb067058dec17b0f5596aca9296`，
实测server为`PostgreSQL 18.4 (Debian 18.4-1.pgdg12+1)`、Linux amd64。

## 已验证范围

Docker构建权威分别执行两条互补链路：

1. `abi-runtime-gate`把module和官方C probe装入精确runtime，通过
   `load_external_function()`解析`_PG_oauth_validator_module_init`，校验module
   magic、OAuth magic、callback表、startup/shutdown、PostgreSQL allocator、panic、
   PostgreSQL ERROR、null输入和错误major；SQL probe返回`t`。
2. `pgx-oauth-gate`使用PostgreSQL 18公开的`PQsetAuthDataHook`发起真实libpq SASL
   `OAUTHBEARER`连接。Gate只在临时test image中生成120秒ES256合成JWT，验证HBA
   issuer/scope、requested role、callback授权、allocator返回的`authn_id`以及
   `system_user`；随后把签名段篡改的token沿同一wire path提交并确认拒绝。

成功连接内执行的断言为：

| 字段 | 实测值 |
| --- | --- |
| `server_version_num` | `180004` |
| `current_user` | `gomtm_candidate_ordinary` |
| `system_user` | `oauth:pggomtm:v1;u=usr_pgx_gate;actor=client:cli_pgx_gate;d=dlg_pgx_gate;m=oauth;a=1;p=ordinary` |
| 有效token | 登录成功 |
| 篡改token | `OAuth bearer authentication failed`，登录拒绝 |

libpq在调用自定义hook前先用空bearer取得server提供的issuer与scope，因此server日志中
会出现一次预期的参数探测失败；随后同一连接流程提交有效token并成功。这与PostgreSQL
18上游`oauth_hook_client`测试顺序一致，不是validator误拒绝成功fixture。

## 复验命令与结果

真实OAuth target首次运行时，`cargo fmt --check`精确指出新Rust fixture helper的一处
格式差异并以exit 1停止，尚未进入runtime。该失败在同一中间image稳定复现；只应用
rustfmt给出的换行后，聚焦格式检查exit 0。随后执行：

```bash
docker build --progress=plain \
  --target pgx-oauth-gate \
  --tag mtmpg-task44-oauth-smoke:local .
```

结果为exit 0。日志明确包含以下不含token的结果：

```text
PG18.4 OAuth allow, role and system_user smoke passed
PG18.4 OAuth tampered-token rejection smoke passed
```

tracked内容原样提交为`696a1cb9dc3e2416726c7cd88fbdaa68e6293c76`后，从该干净
工作树执行默认target：

```bash
docker build --progress=plain \
  --tag mtmpg-task44-final:696a1cb .
```

结果为exit 0。该构建通过内容寻址cache复用刚实际执行成功的相同OAuth layer和此前已
实际执行成功且输入未改变的ABI runtime layer，并在final stage同时要求两枚成功sentinel；
任一gate缺失均无法生成最终image。

本地取证image没有registry digest，不能作为部署身份：

两张image的`RepoDigests=[]`、architecture为`amd64`、OS为`linux`。

## 临时fixture与最终image隔离

- 合成signing fixture只存在于`pgx-oauth-gate`测试源码；临时JWT以`0600`创建，测试
  helper不打印token内容。
- OAuth client错误检查会拒绝把完整token反射到client错误；PostgreSQL核心和module
  日志均未打印token。
- Gate完成后删除有效/篡改JWT、token generator、libpq smoke client和临时PG data。
- Gate target文件系统只保留测试module与成功sentinel；检查确认上述临时文件均不存在。
- Final image检查确认不存在测试module、ABI probe module、两枚sentinel、helper、JWT或
  PG data，只保留无gate的`/usr/lib/postgresql/18/lib/pggomtm.so`。
- Final `.so`继续通过内置JWKS、gate identity和allocator probe marker扫描，且没有
  `libcurl`动态依赖。

## 结论与限制

本次唯一可声明的已验证minor是**PostgreSQL 18.4**。Runtime代码按PG18 major接受
stable line，不等于18.5或未来minor已获部署批准；每个新minor仍必须重新构建、运行
同一真实loader/OAuth矩阵并发布新的不可变digest。

本证据也不表示production validator已经完成。OAuth成功路径使用明确的
`pgx-oauth-gate`测试feature和合成public JWKS；当前无gate final callback仍默认拒绝
所有token。外部只读config/JWKS、正式callback、release manifest、OCI registry digest
和gomtmui candidate E2E仍属于后续OpenSpec任务。
